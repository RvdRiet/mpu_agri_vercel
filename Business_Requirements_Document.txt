EXECUTIVE SUMMARY
================================================================================

The Mpumalanga AgriSupport platform is a multi-channel (web, iOS, Android)
digital system that provides farmers, agri-applicants, and departmental staff
in Mpumalanga with:

  (a)  A comprehensive advisory and decision-support hub — advisory guides
       covering crop, livestock, and environmental farming; real-time weather
       data; farming alerts; NDA Price Watch market information; and an
       AI-assisted chatbot with calculators, guided diagnoses, emergency
       triage, and seasonal planning tools.

  (b)  A digitised grant application and tracking workflow — multi-step
       application forms for Crop, Livestock, Environmental, and Masibuyele
       Esibayeni (MESP) 2025/26 programmes, with autosave, commodity-aware
       dynamic form logic, inline biosecurity planning, health checklists,
       document uploads, and POPIA-compliant consent.

  (c)  A staff portal for application review — MESP procedure-stage
       progression with mandatory evidence capture, additional documentation
       requests, application approval/rejection, PDF export, and support
       issue management.

  (d)  Secure, scalable architecture — a single Vercel serverless API as the
       sole gateway to Azure PostgreSQL (structured data), Azure Blob Storage
       (document files), WinSMS (2FA), OpenWeather, and OpenStreetMap/Nominatim.
       Web, iOS, and Android clients call only this API. Mobile clients use
       local SQLite for offline caching and draft storage.

This BRD is the authoritative specification of business and functional
requirements. It is intended for project sponsors, business analysts, solution
architects, developers, testers, and compliance officers.

--------------------------------------------------------------------------------

IN SCOPE:

  1.  Public-facing website (HTML, CSS, JavaScript) encompassing:
      - Home page with programme overview, scroll-reveal sections, and chatbot.
      - Programme pages: Crop Management, Livestock Management (Masibuyele
        Esibayeni), Environmental Enrichment.
      - Weather dashboard with real-time data for 16 Mpumalanga locations.
      - Farming alerts aggregated from multiple RSS feeds with category
        filtering and Mpumalanga relevance scoring.
      - Advisory guides: crop, livestock, environmental — detailed
        species-by-species encyclopaedic content.
      - E-learning resources and downloadable materials.
      - NDA Price Watch integration (latest PDF resolution via serverless
        function or direct link).
      - Support page with emergency contacts, tabbed service directory, and
        login-gated issue reporting.

  2.  Grant application forms — four multi-step wizards:
      a. Crop Grant (5 steps)
      b. Livestock Grant (5 steps) — official MESP 2023/24 form, fully
         digitised with all Sections A–G, commodity-aware dynamic logic,
         biosecurity plan template, stock health checklists, health record
         uploads, declaration, and document checklist.
      c. Environmental Grant (5 steps)
      d. Masibuyele Esibayeni 2025/26 (6 steps, Sections A–G)

  3.  Application tracking page for applicants — status, progress bar,
      procedure stage history, reviewer messages, document request notices.

  4.  Applicant authentication — SA ID number and password, with optional
      SMS-based 2FA via WinSMS.

  5.  User profiles and document management — personal details, farm profile,
      document uploads (ID copies, proof of land, business plans, health
      records, biosecurity plans, etc.).

  6.  Staff portal — application queue, full detail view with PDF export,
      MESP procedure-stage engine with evidence capture, additional
      documentation requests, approve/reject workflow, and reported-issue
      management.

  7.  AgriSupport Chatbot — rule-based knowledge assistant with:
      - 200+ topic entries across crop, livestock, environmental, legal,
        support, and practical farming domains.
      - Interactive guided diagnosis flows (maize pest, small-stock parasite,
        poultry mortality) with structured Field Report Card output.
      - Quick calculators (fertiliser, stocking rate).
      - Emergency triage for livestock and crop health crises.
      - Dynamic planners (weekly, seasonal, startup, outbreak).
      - Session context memory (location, enterprise).

  9.  Support issue reporting — logged-in users can report issues (website

      dropdowns; staff manage issue lifecycle (Submitted → In Progress →

  10. Feedback widget — star rating and text feedback from any page that
      includes the module.

  11. Backend API (Vercel serverless) — sole component communicating with
      Azure PostgreSQL, Azure Blob Storage, WinSMS, OpenWeather, and map/
      geocoding services.

  12. Mobile applications (iOS, Android) — native or cross-platform, with
      offline SQLite caching and draft storage, syncing via the same API.

2.  BUSINESS BACKGROUND

--------------------------------------------------------------------------------
2.1  Business Problem
--------------------------------------------------------------------------------

Farmers and applicants in Mpumalanga lack a single, accessible digital platform
to access farming advice, weather data, market information, and departmental
grant applications. Manual paper-based application processes cause delays, data
loss, and inconsistent decision-making. The department requires a digital staff
portal for auditable, stage-gated application review aligned with the MESP
Procedure Manual.

--------------------------------------------------------------------------------
2.2  Business Drivers and Goals
--------------------------------------------------------------------------------

  BG-1:  Improve farmer access to information, advisory content, and grant
         applications across web and mobile channels.
  BG-2:  Digitise the official MESP application form to reduce manual handling,
         improve data quality, and enforce business rules at point of capture.
  BG-3:  Reduce application processing time through stage-gated workflow with
         mandatory evidence capture at each transition.
  BG-4:  Centralise authentication, profile, and document storage in a secure,
         scalable backend (Azure) accessed only via a single API gateway.
  BG-5:  Ensure compliance with POPIA, WASPA, and relevant agricultural
         legislation (Animal Diseases Act, NEMBA, etc.).
  BG-6:  Enable future scalability (additional programmes, users, or channels)
         without changing core architecture.

--------------------------------------------------------------------------------
2.3  Current State (As-Is)
--------------------------------------------------------------------------------

  - A fully functional static website exists with advisory guides (crop,
    livestock, environmental — encyclopaedic depth), e-learning, weather
    dashboard, farming alerts, NDA Price Watch, support contacts, and an
    AI-assisted chatbot with guided diagnostics and calculators.
  - Four multi-step grant application forms are operational (Crop, Livestock,
    Environmental, MESP 2025/26) with autosave drafts and client-side
    validation.
  - The Livestock Grant form has been fully digitised from the official MESP
    2023/24 PDF, implementing all Sections A–G with commodity-aware dynamic
    form logic, inline biosecurity plan template (10 sections, DALRRD-aligned),
    commodity-specific stock health checklists, health/vaccination record
    uploads, formal declaration, and category-conditional document checklist.
  - A staff portal provides application review, MESP procedure-stage
    progression with mandatory structured evidence, additional documentation
    requests, PDF export, and support issue management.
  - An applicant tracking page shows application status, progress bar,
    procedure stage history, and reviewer messages.
  - Authentication and data persistence are currently client-side
    (localStorage/sessionStorage) with optional Firebase integration.
  - There is no central backend database or server-side document store in
    production; no live server-side 2FA or SMS integration.

--------------------------------------------------------------------------------
2.4  Desired Future State (To-Be)
--------------------------------------------------------------------------------

  - All persistent data stored in Azure PostgreSQL; document files in Azure
    Blob Storage.
  - A single Vercel serverless API as the sole gateway to Azure, WinSMS,
    OpenWeather, and map services.
  - Applicant sign-in via SA ID and password with optional SMS 2FA (WinSMS).
  - Staff sign-in with username and password; full MESP procedure-stage
    workflow with auditable evidence and decision records in PostgreSQL.
  - Mobile apps (iOS, Android) with offline SQLite caching and API sync.
  - All existing client-side features preserved and enhanced with server-side
    persistence and security.

  BO-2:  Digitise the official MESP application form with all sections,

  BO-3:  Enable staff to review and decide on applications with a full audit

  BO-4:  Store all data in Azure (PostgreSQL + Blob) with access only through

  SC-4:  All client-facing features (chatbot, weather, alerts, guides) are

5.  FUNCTIONAL REQUIREMENTS

Requirements use "shall" language. Each is assigned a unique ID (FR-xxx) for
traceability. Acceptance criteria follow each group.

--------------------------------------------------------------------------------
5.1  User Registration and Authentication (FR-001 to FR-004)
--------------------------------------------------------------------------------

FR-001:  The system shall allow applicant registration using SA ID number
         (13 digits, validated) and password. Credentials stored in Azure
         PostgreSQL; access only via the Vercel API.

FR-002:  The system shall allow sign-in using SA ID and password. On success
         with 2FA enabled, the API shall generate a short-lived OTP, store
         it hashed with expiry, and send via WinSMS. Clients never call
         WinSMS directly.

FR-003:  After OTP verification, the API shall issue a session (JWT or
         secure cookie). Protected resources accessible only after full
         authentication.

FR-004:  Password policy (minimum length, complexity) and reset flows shall
         be configurable per business rules.

Acceptance: Registration and sign-in work on web and mobile; 2FA blocks access
until correct OTP; no client holds database or WinSMS credentials.

--------------------------------------------------------------------------------
5.2  Two-Factor Authentication via SMS (FR-005 to FR-007)
--------------------------------------------------------------------------------

FR-005:  2FA shall be optional and configurable. When enabled, the backend
         generates a numeric OTP, stores the hash and expiry, and calls
         WinSMS (E.164 format).

FR-006:  Client sends OTP to the API for verification. API validates and
         issues session only on success within the expiry window.

FR-007:  WinSMS integration shall exist only in Vercel serverless functions.
         WASPA and opt-out compliance per WinSMS and department policy.

Acceptance: SMS sent only by backend; sign-in blocked without valid OTP.

--------------------------------------------------------------------------------
5.3  User Profile and Farm Profile (FR-008)
--------------------------------------------------------------------------------

FR-008:  Applicants can view and update:
         - Personal: name, contact, email, address, SA ID (read-only).
         - Farm: name, district, GPS coordinates, total size (ha), soil
           type, commodities, irrigation type, previous yields.
         - Documents: replace or upload ID copy, proof of ownership, bank
           details.
         All persisted in Azure PostgreSQL via the API.

Acceptance: Profile data persists across sessions; accessible from web/mobile.

--------------------------------------------------------------------------------
5.4  Document Upload, Storage, and Retrieval (FR-009 to FR-010)
--------------------------------------------------------------------------------

FR-009:  Applicants upload documents via the API. API stores bytes in Azure
         Blob and metadata (user_id, type, filename, content_type, size,
         blob_path, uploaded_at) in PostgreSQL.

FR-010:  Download/view via API only (fetch from Blob or return signed URL).
         No client direct Blob access.

Acceptance: Upload/download works; no client-side Blob credentials.

--------------------------------------------------------------------------------
5.5  Grant Applications — General (FR-011)
--------------------------------------------------------------------------------

FR-011:  The system shall support four grant application programmes:
         (a) Crop Grant — 5-step wizard
         (b) Livestock Grant — 5-step wizard (official MESP 2023/24 form)
         (c) Environmental Grant — 5-step wizard
         (d) Masibuyele Esibayeni 2025/26 — 6-step wizard

         Form data stored in PostgreSQL; file attachments via Blob. Status
         values: Draft, Submitted, Under Review, Additional Documentation
         Required, Profiling and Shortlisting, Farm Assessment, HOD Approval,
         Induction, Procurement, Pre and Post Delivery Inspection, Monitoring
         Evaluation and Advisory, Approved, Rejected.

Acceptance: All four forms submittable; status visible to applicant and staff.

--------------------------------------------------------------------------------
5.6  Livestock Grant Application — Official MESP Form (FR-030 to FR-042)
--------------------------------------------------------------------------------

FR-030:  The livestock grant form shall be a complete digital version of the
         official MESP Livestock Improvement Programme Application Form
         (2023/24), structured as a 5-step wizard:

         Step 1 — Livestock Selection:
           - Commodity (Beef Cattle, Dairy Cattle, Goats, Sheep, Pigs,
             Poultry, Rabbits, Fish)
           - Preferred breed (1st and 2nd choice, filtered by commodity)
           - Farmer category (Subsistence, Small Holder, Commercial)
           - Previous MESP benefit: yes/no, prior commodity, commodity
             returned (yes/no), details with date
           - Farmer category definitions (official text)

         Step 2 / Section A — Project Details:
           - Legal entity type: Sole Proprietor, Cooperative, CPA, Trust,
             PTY Ltd, Other (specify)
           - Registered name / applicant name (required)
           - Registration number (attach copy)
           - Representative name and SA ID (13 digits)
           - Telephone, cell, fax
           - Applying as group or individual
           - Physical/business address with OpenStreetMap geocoding
             (Leaflet map, Nominatim), GPS coordinate auto-fill, and
             "Use current location" (device geolocation)
           - Postal address
           - Optional email and preferred communication method (Phone,
             SMS, Email, WhatsApp checkboxes)
           - Total project members with demographic breakdown: Men, Women,
             Youth, People with Disability

         Step 3 / Section B — Farm Details:
           - Farm name (as on title deed)
           - Land breakdown: total (ha), natural grazing (ha), planted
             pasture (ha), arable (ha), plantations (ha)
           - District (Ehlanzeni, Gert Sibande, Nkangala — dropdown)
           - Local municipality (dynamic filter based on district)
           - Area name, ward number
           - GPS coordinates (latitude, longitude)
           - Type of land access: Own, Lease, Communal, PTO, RTO, Other
           - Lease period (years) — conditional; minimum 5-year validation

         Step 4 / Sections C–G — Infrastructure & Stock:
           (See FR-031 to FR-037 for detailed specification)

         Step 5 — Documents & Submit:
           (See FR-038 to FR-042 for detailed specification)

FR-031:  Section C — Farm Infrastructure (commodity-aware):
         - Fencing and grazing camps: Beef Cattle, Dairy Cattle, Goats, Sheep
           (boundary fence, grazing camps, condition — Excellent/Good/Fair/Bad)
         - Water sources: all commodities (river, streams, windmill, borehole,
           other — specify)
         - Handling facilities: kraals, crush pens (Beef Cattle, Dairy Cattle,
           Goats, Sheep, Pigs); dipping facilities (ruminants only); condition
           of kraals (Excellent/Good/Fair/Bad)
         - Comments text field
         - When commodity changes, all hidden fields auto-clear.

FR-032:  Section D — Livestock Housing (Poultry, Pigs, Rabbits only):
         - Capacity of houses
         - Condition of housing (Excellent/Good/Fair/Bad)
         - Feeders and drinkers (Yes/No)
         - Heating available (Yes/No)
         - Brief description of housing
         - Steel cages for layers (Poultry only; Yes/No/N/A)
         - Steel cages for rabbits (Rabbits only; Yes/No)
         - Cage description with dimensions
         - Feed storage facilities (Yes/No)
         - Condition of storage (Excellent/Good/Fair/Bad)
         - Biosecurity plan available (Yes/No)

FR-033:  Section E — Dairy Infrastructure (Dairy Cattle, Goats only):
         - Dairy machinery and facilities details
         - Milking capacity (litres/day)
         - Storage tank size (litres/kilolitres)
         - Milk sold type (Raw / Processed)
         - Planted pasture type and hectares
         - Available feeds: type and quantities

FR-034:  Section F — Aquaculture Infrastructure (Fish only):
         - Dams, ponds, and facilities details
         - Production capacity (tons per annum)
         - Fish sold type (Unprocessed / Processed)
         - Markets
         - Feed storage facilities

FR-035:  Section G — Current Number of Stock:
         - Stock numbers for selected commodity only:
           * Beef Cattle: Breeding Bulls, Steers, Cows/Heifers
           * Dairy Cattle: Breeding Bulls, Cows/Heifers
           * Goats: Bucks, Does
           * Sheep: Rams, Ewes
           * Pigs: Boars, Sows
           * Rabbits: Bucks, Does
           * Poultry: Broilers (per 6-week cycle), Layers, Free Range
           * Fish: Current stock (estimated number)
         - Official notice: "Health records of current stock to be made
           available when requested by the department, especially vaccination
           history, TB and Brucellosis test results."
         - Quarantine infrastructure (Yes/No) — all commodities
         - Brand mark (Yes/No, attach copy) — cattle, goats, sheep, pigs
         - Additional information text field

FR-036:  Stock Health Checklist (commodity-specific, Yes/No/N/A radios):

         Beef/Dairy Cattle (10 items):
           TB test (12 months), Brucellosis test (12 months), Brucellosis
           vaccination (S19/RB51), Blackquarter, Anthrax, Lumpy Skin Disease,
           Botulism, dipping (external parasites), internal parasite dosing,
           Rift Valley Fever.

         Goats (6 items):
           Pulpy kidney, Pasteurella, Bluetongue, internal parasites
           (wireworm), external parasites, Heartwater.

         Sheep (6 items):
           Pulpy kidney, Bluetongue, Pasteurella, shearing, internal parasites,
           external parasites.

         Pigs (6 items):
           Iron injection (piglets), Erysipelas, deworming (sows), mange,
           ASF wildlife barrier, no swill feeding.

         Poultry (7 items):
           Newcastle Disease, Gumboro, Marek's, Infectious Bronchitis,
           coccidiosis, deworming, all-in-all-out management.

         Rabbits (4 items):
           Coccidiosis, mange, pasteurellosis (snuffles), cage sanitation.

         Fish (4 items):
           Water quality testing, quarantine, disease monitoring, escape
           prevention.

         Plus: Additional health notes text field.

FR-037:  Health and Vaccination Records Upload (commodity-specific):
         - TB test certificate (Beef/Dairy Cattle)
         - Brucellosis test certificate (Beef/Dairy Cattle)
         - Vaccination record/card (all commodities)
         - Dipping/dosing record (cattle, goats, sheep, pigs)
         - Veterinary inspection report (all, optional)
         - Other health documents (all, optional)
         Accepted formats: PDF, JPG, PNG.

FR-038:  Biosecurity Plan (Step 5) — two modes:
         Mode A: Upload existing biosecurity plan (PDF/DOC).
         Mode B: Complete inline template (10 sections, DALRRD/DAFF-aligned):
           1. Farm Identification (auto-fills farm name)
           2. Access Control & Visitor Management
           3. Quarantine & Isolation Procedures
           4. Animal Health & Veterinary Programme (commodity-specific):

7.  PROCESSES AND WORKFLOWS

--------------------------------------------------------------------------------
7.1  Applicant Registration and Sign-In
--------------------------------------------------------------------------------

  1. User enters SA ID and password on client.
  2. Client sends credentials to Vercel API (POST /api/auth/login).
  3. API validates against PostgreSQL; returns 401 if invalid.
  4. If 2FA enabled: API generates OTP → stores hash + expiry → calls
     WinSMS → returns pending2FAId. Client shows "Enter code" screen.
  5. User enters OTP; client sends to API (POST /api/auth/verify-2fa).
  6. API verifies OTP + expiry → issues JWT/session → returns success.
  7. If 2FA not enabled: session issued after step 3.

--------------------------------------------------------------------------------
7.2  Grant Application Submission
--------------------------------------------------------------------------------

  1. Applicant signs in. Selects programme (crop/livestock/environmental/MESP).
  2. Completes multi-step wizard. Autosave captures state to localStorage
     on each input (debounced).
  3. On submit: client sends payload + file uploads to Vercel API.
  4. API validates (business rules, category compliance, lease period, etc.);
     writes form data to PostgreSQL; uploads files to Blob.
  5. API returns application ID and status (Submitted).
  6. Client clears autosave draft; shows confirmation and tracking link.

--------------------------------------------------------------------------------
7.3  Livestock Grant — Commodity-Aware Form Flow
--------------------------------------------------------------------------------

  1. Applicant selects commodity → breeds filter, infrastructure sections
     show/hide, stock numbers filter, health checklist adapts.
  2. Step 4 displays only relevant sections:
     a. Section C: fencing for ruminants, water for all, kraals for large
        livestock, dipping for ruminants.
     b. Section D: housing for poultry/pigs/rabbits (with steel cages,
        feed storage, heating, biosecurity indicator).
     c. Section E: dairy infrastructure for dairy cattle/goats.
     d. Section F: aquaculture infrastructure for fish.
     e. Section G: stock numbers, health checklist, health record uploads
        — all for selected commodity only.
  3. Step 5 shows only commodity-relevant document uploads and biosecurity
     sections.
  4. Checklist table dims N/A rows based on farmer category.
  5. On commodity change: all hidden fields auto-clear to prevent stale data.

--------------------------------------------------------------------------------
7.4  Staff Review and Decision
--------------------------------------------------------------------------------

  1. Staff signs in to portal; views application queue.
  2. Selects application; views full detail (all Sections A–G, health
     checklists, biosecurity, declaration, document checklist).
  3. May request additional documentation with reviewer message.
  4. May advance through MESP procedure stages (Submitted → ... → Approved)
     with mandatory evidence capture at each transition.
  5. May approve or reject with audit record.
  6. May export application as PDF (jsPDF).

--------------------------------------------------------------------------------
7.5  MESP Submission Compliance Validation
--------------------------------------------------------------------------------

  1. On submit: system validates eligibility declarations, category-specific
     checklist, and land access rules.
  2. If non-compliant: submission blocked with specific error messages.
  3. If compliant: application created as Submitted.

--------------------------------------------------------------------------------
7.6  Support Issue Lifecycle
--------------------------------------------------------------------------------

  1. Logged-in user submits issue (category + description) from Support page.
  2. Issue stored and displayed in staff issue queue.
  3. Staff updates status: Submitted → In Progress → Resolved.

--------------------------------------------------------------------------------
7.7  Weather and Map Data
--------------------------------------------------------------------------------

  1. Client requests weather/map data from API endpoint.
  2. API calls OpenWeather/WeatherAPI or Nominatim with server-held key.
  3. Returns processed response to client. Client never sees API key.
